I read the full repo (myself plus a 44-agent parallel review workflow whose critical findings were each adversarially verified against the code), then patched 24 files. Everything passes npm run typecheck, npm run lint, and npm run build, and I smoke-tested the real game in a browser: created a new world, watched terrain generate and mesh through the new worker pool, confirmed zero console errors, and confirmed auto-save now fires on schedule. The docs in docs/claude-upgrades/ turned out to be substantially aspirational — the worker pool, buffer pooling, and real greedy meshing they describe were not implemented, so I treated code as the only source of truth.

Performance (the big wins)
Worker pool — WorldManager.ts used a single worker for both terrain gen and meshing, so its "8 jobs in flight" all serialized on one thread. The worker was already stateless, so I built a real pool (2–4 workers by core count, round-robin, ticket protocol unchanged). Chunk streaming throughput now actually scales.
Block-edit lighting flood rewrite — every block place/break (and every fluid level change) ran a flood-fill making ~370k getBlock/getLight/setLight calls, each allocating an object plus a key string — roughly 800k temporary allocations per edit, on the main thread. lighting.ts now uses cached typed-array access throughout; same output, near-zero allocation.
Chunk-load light reconciliation — border reconciliation seeded tens of thousands of provably no-op BFS entries per chunk (both sides fully sunlit). Now skipped, which pairs with a gen-time fix below that makes the skip safe.
Mesher — geometry.ts iterated all 384 Y levels per chunk and allocated 768 scratch arrays per mesh; it now computes the occupied Y range once and reuses a hoisted buffer. Its fixed scratch buffers also cost 60MB per importing context (now ×4 workers!) — they now start at ~5MB and grow on demand, which also removes a silent face-drop at the old 100k-face cap.
Chunk-gen noise caching — climate parameters were recomputed 10–20× per column (terrain pass, biome pass, beach probes, and again across the whole 32×32 padded tree pass). Pure memoization scoped to a single generateChunk call (biomes.ts, chunkGeneration.ts) — bit-identical output, so seeds and saves are unaffected. Also hoisted the per-chunk 786KB light-queue allocation.
React re-render storms killed: DayNightCycle was calling App-level setState every frame with values nobody read (App-wide re-render through sunrise/sunset); the mining breakingVisual re-rendered the whole App per frame (now an imperative mesh fed by a store); GameLoop pushed a continuously-decaying saturation float through React 20×/s (now quantized + change-gated); the drop-magnet physics and DayNightCycle allocated ~25 objects per frame (hoisted to scratch/constants).
Main-thread geometry copies — Float32BufferAttribute copies its input array; ChunkMesh.tsx now wraps the worker-transferred buffers directly.
Smaller: shadow maps no longer render for the sun/moon light that's at zero intensity (halves shadow cost), aurora update skips entirely in daylight, getOpacity is now a flat table lookup in the lighting/AO hot loops, the FPS limiter no longer chronically undershoots, and the menu panorama no longer rebuilds its entire WebGLRenderer on every resize event.
Bugs fixed
Ghost chunks: a remesh landing during a chunk's fade-out disposed live geometries out from under React, stranding a frozen translucent chunk forever (and leaking its GPU buffers). The fade now completes untouched.
Auto-save was starved: the 10s timer reset every time inventory/health/breath changed, so it basically only saved while idle. Fixed with a latest-ref interval plus a best-effort save on window close — previously a crash lost everything since your last pause.
Fluids: pouring fluid onto a source pool destroyed the source (breaking infinite water and lava/obsidian rules), and flowing water/lava never receded after its source was removed. Both fixed in fluids.ts with the standard drain algorithm.
Saplings effectively never grew: 3 random column probes per 3s across all loaded chunks ≈ 114 hours per sapling, and a hardcoded y 64–199 scan window. Now probes per-chunk over the full world column (~half an hour to grow); the staged metadata behavior is preserved exactly.
Dark cave mouths: gen-time skylight never spread horizontally from sunlit air, leaving breach shafts and cliff notches pitch black except near chunk borders. Sunlit boundary cells are now seeded into the gen BFS.
Bed wedging: landing on a bed snapped the player half a block into it, blocking all movement. The downward collision snap now uses the actual supporting block's height.
World-gen border artifacts: vegetation had a 3-block dead margin on every chunk edge (visible 16-aligned barren grid), and trees used an order-dependent in-chunk soil check that produced orphan floating canopies across borders. Both are now consistent and deterministic.
Smaller: glass/ice/wool/snow/bed blocks were completely silent (added block.generic.* fallback), dirt played sand sounds, placing a block could silently delete torches/plants (they now pop off as drops), shift-click mutated React state objects in place, and hunger damage was frame-rate-dependent in the game loop. Deleted the dead gen.worker.ts/mesh.worker.ts.
What still needs follow-up
Entity/combat system: there is none. Player, drops, and particles are all that exist — the boss-killing direction needs an entity foundation (shared transform/health/AI tick, server-authoritative-friendly state) before anything else. This is the single biggest architectural gap.
Real greedy meshing: the current "greedy" pass computes merge rectangles but still emits one quad per cell — zero vertex reduction. Doing it properly needs shader-side UV tiling within atlas tiles and AO-compatible merge rules.
Mesh job payloads: each mesh job still structured-clones ~1MB to a worker; a worker-side chunk cache would eliminate it (the round-robin pool makes this slightly more involved).
App.tsx decomposition: still a ~2,300-line god component; chat/command state at the root re-renders the app per keystroke. Dimension/multiplayer readiness wants world state pulled fully out of React.
Lower priority, catalogued during review: per-chunk material cloning (3 materials/chunk even with fade off), HRTF panners on every footstep, per-chunk IndexedDB transactions in the save loop, and crafting shape-matching losing grid width (rotated recipes match).

Twelve FPS at render distance 48 has identifiable causes, and most of them are fixable right now. At RD 48 you have ~7,200 chunk columns mounted, which means: ~7,200 useFrame callbacks every frame (one per chunk, almost all idle), ~21,000 cloned materials forcing full uniform re-uploads on every draw call, ~29,000 scene nodes recomputing world matrices every frame, the block-targeting raycaster testing the entire scene per frame, geometry for every cave face underground (invisible from outside, but uploaded and vertex-processed anyway — likely 1–2GB+ of GPU data, which is what really tanks FPS), and CPU-side copies of all of it doubling RAM.

My plan, in impact order: (1) dark-face culling for far chunks — enclosed cave faces are only ever visible when you're inside them, so chunks beyond a few chunks of distance skip fully-unlit faces and remesh on approach; (2) shared chunk materials + one global fade ticker instead of per-chunk clones and per-chunk useFrame; (3) freeze static chunk matrices; (4) replace the whole-scene raycast with voxel DDA stepping; (5) release CPU-side geometry copies after GPU upload; (6) stop plant growth from scanning all 7,200 chunks.
